## Problem Statement

家长想给自己孩子持续记录身高/体重/头围/BMI，按 WHO 和中国卫健委标准看百分位曲线，并希望随手把数据分享给亲友或医生。

GitHub 上同类 OSS 13+ 个，iOS/小程序数十款，但仍有明显缺口：

- **国际 App / OSS（GrowthKit / Growth / Child Growth Tracker / WHO anthro / pygrowthstandards / pchart）不支持中国卫健委 2023/2025 标准**——而这是中国家长的合规和习惯锚点
- **国产 App 大多是综合育儿工具**，曲线藏在三级菜单（少数派评测原话："层层翻找"），且广告和隐私政策不透明
- **没有原生 TS/JS 库 + 浏览器 UI + AI agent（MCP）一体化的方案**——TS 生态目前最像样的 `pchart` 是 GPL 仅渲染、`pygrowthstandards` 是 Python；MCP 接入完全空白
- **没有许可宽松（MIT/Apache）的 npm 计算库**——商业项目想集成只能自己把 LMS 算法重写一遍

groowooth 的路径："先给自己孩子用 → 推广给身边亲友 → 开源给 OSS 社区"。**v1 即把 AI agent 作为一等公民**——这是没有竞品在做的窗口。

## What Makes This Cool

1. **唯一 v1 即支持 MCP 的成长曲线工具**——AI agent 浪潮下，把"成长曲线"做成 LLM 可调用工具的第一个；可被 Claude Desktop / Cursor / Cline 一键接入
2. **首个把卫健委 2023/2025 + WHO 双轨打包进 npm 的库**——TS/JS 社区想用中国标准，目前必须从 PDF 自己抠
3. **Web + 本地存储 + 开源**——分享 URL 即可推广，零安装、零账号、零隐私焦虑、零运维
4. **专注一件事**——不做喂奶、睡眠、辅食、疫苗、社区、关怀文章；只做曲线

## Constraints

- **平台**：网页（v1）；MCP server 同步发布
- **目标用户**：自己 → 中文亲友 → OSS 社区开发者（次要）
- **必须**：支持中国卫健委 2023 / 2025 最新标准；离线可用；中文为主，英文摘要
- **不做（v1 显式 out-of-scope）**：账号系统、云同步、付费、广告、聊天界面、综合育儿功能、临床诊断/医疗建议
- **足月儿假设**：v1 仅支持足月儿（≥37 周）；早产儿矫正与 Fenton 2013 标准列入 v1.1 roadmap，输入边界给出明确警告
- **协议**：源码 MIT 或 Apache-2.0；标准数据公有 → 可自由打包

## Premises (经用户确认)

1. **三层架构**：headless TS core + web UI + MCP server，monorepo（pnpm workspaces）
2. **中国卫健委 + WHO 双轨切换**是核心差异化
3. **只做曲线一件事**——不做综合育儿
4. **本地优先**：web 用 IndexedDB；MCP 走 stateless 计算 API（数据由调用方传，服务端 0 儿童数据）
5. **免费 / 无广告 / 开源**（MIT 或 Apache-2.0）
6. **AI agent 一等公民**：v1 即含 MCP；不是 v2

## Approaches Considered

### Approach A: MVP 优先（先 web，MCP 留 v2）
- 范围：core lib + 极简 web。MCP 二期。
- 工作量：S（≈1 个周末 / 8h）
- 风险：低
- Pros：最快 ship；先验证家长是否持续用
- Cons：错过 v1 即"AI agent 友好"的传播窗口；core 接口在没有 MCP 压力下设计，后期回填要 refactor
- 不选。

### Approach B: 三层齐发（chosen）
- 范围：core + web + MCP server **同时发布**，pnpm workspaces monorepo
- 工作量：M-L（**3–4 个周末 / 30–40h**，已含 NHC PDF 数字化攻关 + R 黄金值校准 + 两包 CI/CD + web 部署设置）
- 风险：中
- Pros：
  - core 反正必须独立（MCP 强制要求），多写的 MCP 胶水仅 ~200 行
  - 三个标签"开源 + 中国标准 + AI 友好"v1 同时立住
  - 能上 awesome-mcp 列表吃早期红利
  - core 接口被 web + MCP 两个 client 同时使用 → 设计质量更高
- Cons：要管 2 个 npm 包（core + mcp）+ 1 个 web 部署的版本同步；要写 MCP 接入文档
- Reuses：pchart 的 WHO LMS JSON 仅参考组织方式（GPL，**不抄数据/代码**）；pygrowthstandards 算法逻辑（MIT，可对照）；`@modelcontextprotocol/sdk`

### Approach C: Agent-first 横切
- 范围：把"和 AI 对话来记录/查看"做成主交互
- 工作量：M-L
- 风险：高（UX 赌局：家长真愿意聊天吗？少数派评测核心痛点是"打开就看图"，不是"和 AI 聊"）
- 不选。可作为 B 的 v2 增量加。

## Recommended Approach

**Approach B：三层 monorepo 一次发布**

### 技术栈

| 层 | 技术 | 理由 |
|----|------|------|
| 包管理 | pnpm + workspaces | 最轻；npm publish 友好 |
| 构建 | tsup（core, mcp）/ Vite（web） | tsup 出 ESM+CJS 双产物；Vite 静态部署 |
| core deps | 零 runtime（算法纯函数 + zod for input validation） | bundle size 极小；任何环境跑 |
| web | React + TypeScript + idb | 团队最熟；idb 是 IndexedDB 的轻量封装 |
| 图表 | **手写 SVG（在 core 中）**；后续可换 uPlot | 5 条百分位线 + 数据点 + axis label，复杂度低；零依赖；web 和 mcp 共享同一份逻辑（避免重复） |
| MCP | `@modelcontextprotocol/sdk` (Node 18+, stdio transport) | 官方 SDK；stdio = Claude Desktop 默认 |
| 测试 | Vitest（core 100%，含 golden 对照）+ Playwright（web 关键路径） | 标准选择 |
| CI | GitHub Actions（Node + headless Chrome smoke） | 免费；core 双环境验证 |
| 部署 | Cloudflare Pages（web） | 免费 + 自定义域名 + 全球 CDN |
| 域名 | `groowooth.app` (v1) | `.cn` ICP 备案 2-4 周不在关键路径，列 v1.1 评估 |

### Core API（v1）

```ts
// @groowooth/core
import { assess, lookup, interpret, renderChart } from '@groowooth/core'

// 1. 单次评估
const result = assess({
  ageMonths: 24,
  sex: 'female',
  heightCm: 86,
  weightKg: 12,
  standard: 'nhc',  // 'who' | 'nhc'，默认 'nhc'
})
// → {
//     heightForAge: { zScore: 0.31, percentile: 62, range: 'within_2_sd' },
//     weightForAge: { zScore: 0.18, percentile: 57, range: 'within_2_sd' },
//     bmiForAge:    { zScore: -0.05, percentile: 48, range: 'within_2_sd' },
//     standard: 'nhc',
//     standardVersion: '2023',
//     disclaimer: '本结果仅为统计参考，不构成医疗建议；如有疑虑请咨询儿科医生。',
//   }

// 2. 拿曲线数据画图
const curves = lookup({
  indicator: 'height-for-age',
  sex: 'female',
  standard: 'nhc',
  ageRangeMonths: [0, 60],
  percentiles: [3, 15, 50, 85, 97],
})
// → { p3: [{age, value}, ...], p50: [...], ... }

// 3. 把单个 z-score 翻译成中文统计描述（不含行动建议）
const text = interpret({
  zScore: 0.31,
  indicator: 'height-for-age',
  sex: 'female',
  ageMonths: 24,
  standard: 'nhc',
})
// → {
//     description: '您家女宝在 24 个月时身高位于第 62 百分位（z=+0.31，按 NHC 2023）。',
//     range: 'within_2_sd',
//     disclaimer: '本结果仅为统计参考，不构成医疗建议；如有疑虑请咨询儿科医生。',
//   }

// 4. 渲染 SVG（web 和 mcp 共用）
const svg: string = renderChart({
  measurements: [{ date: '2026-04-26', ageMonths: 24, value: 86 }],
  indicator: 'height',
  sex: 'female',
  standard: 'nhc',
})
```

**Z-score → 范围标签（v1 用统计语言，不用临床术语）：**
- `'within_2_sd'`：|z| ≤ 2（约 P3-P97）
- `'beyond_2_sd'`：2 < |z| ≤ 3
- `'beyond_3_sd'`：|z| > 3

> **v1 不输出临床诊断词**（"stunted" / "underweight" / "obese" / "矮小" / "肥胖" 等）。
> 这些词在 WHO 标准中按 indicator 不同有特定含义且解读需要医生介入；
> groowooth 定位是"统计可视化 + 数据沉淀"，不是"自动诊断"。
> 每个返回结果都附 `disclaimer` 字段。

### MCP 工具定义（v1）

```ts
// @groowooth/mcp/src/tools.ts
import { z } from 'zod'

export const tools = [
  {
    name: 'assess_growth',
    description: '基于 WHO 或中国卫健委标准评估儿童生长发育，返回各 indicator 的 z-score、百分位、范围标签（统计语言，不含临床诊断）',
    inputSchema: z.object({
      ageMonths: z.number().min(0).max(228),
      sex: z.enum(['male', 'female']),
      heightCm: z.number().min(30).max(250).optional(),
      weightKg: z.number().min(0.5).max(200).optional(),
      headCircumferenceCm: z.number().min(25).max(70).optional(),
      standard: z.enum(['who', 'nhc']).default('nhc'),
    }),
    output: '{ assessments, standard, standardVersion, disclaimer }',
  },
  {
    name: 'get_growth_chart',
    description: '生成 SVG 成长曲线图（含百分位带和孩子数据点）',
    inputSchema: z.object({
      measurements: z.array(z.object({
        date: z.string(),       // ISO date
        ageMonths: z.number().min(0).max(228),
        value: z.number(),
      })),
      indicator: z.enum(['height', 'weight', 'head', 'bmi']),
      sex: z.enum(['male', 'female']),
      standard: z.enum(['who', 'nhc']).default('nhc'),
    }),
    output: 'MCP content blocks: [{ type: "text", text: "<svg>..." }]; v1.1 添加 prefer_format 参数支持返回 PNG base64 image content',
  },
  {
    name: 'interpret_growth',
    description: '把 z-score 翻译成家长友好的中文统计描述（不含行动建议或临床诊断）',
    inputSchema: z.object({
      zScore: z.number(),
      indicator: z.enum(['height-for-age', 'weight-for-age', 'bmi-for-age', 'head-for-age']),
      ageMonths: z.number().min(0).max(228),
      sex: z.enum(['male', 'female']),
      standard: z.enum(['who', 'nhc']).default('nhc'),
    }),
    output: '{ description: 中文统计描述, range: within_2_sd|beyond_2_sd|beyond_3_sd, disclaimer }',
  },
]
```

### Web UI 信息架构

**单页应用，3 个 view：**

1. **Home / 仪表板**
   - 顶部：娃头像 + 名字 + 年龄（年-月）+ 切换娃下拉
   - 主体：4 张曲线缩略卡（身高 / 体重 / 头围 / BMI），点开放大
   - 右下角：「+ 记录」FAB

2. **记录测量（弹窗）**
   - 日期（默认今天）
   - 身高 / 体重 / 头围（任填一项即可保存；BMI 自动算）
   - 异常输入软提示（如"6 个月 80kg"）→ 二次确认
   - 保存即时反馈：z-score + 百分位 + 范围标签 + disclaimer
   - **不弹"建议复测"等提示**——避免焦虑（核心反 pattern）

3. **设置**
   - 切标准（卫健委 ⟷ WHO）
   - 添加 / 切换 / 删除娃；早产儿警告 banner（"v1 假设足月儿"）
   - 导出 JSON / 导入 JSON
   - 深色模式
   - 关于（版本、协议、数据来源声明、GitHub 链接、免责声明）

**坚决不做：登录、推送、订阅、广告、社区、文章 feed、和医生连线。**

### IndexedDB 数据 schema（multi-child）

```
DB: groowooth   version: 1
├── store: children
│   keyPath: id (uuid)
│   value: { id, name, sex, dateOfBirth, gestationalWeeks?, isPreterm?, createdAt, updatedAt }
└── store: measurements
    keyPath: [childId, date]   // composite
    value: { childId, date, ageMonths, heightCm?, weightKg?, headCircumferenceCm?, note? }
    indexes: childId, date
```

**版本迁移**：DB version 跟 web 包 minor 版本同步；schema 变更走 `IDBOpenDBRequest.upgradeneeded`，老数据按 child 维度逐条迁移。导入/导出 JSON 是迁移失败的 fallback。

### 可访问性 + 国际化

- **目标用户包含 60+ 长辈**：tap target ≥44px；正文字号 ≥16px；高对比度暗色主题；不用纯色彩传达信息（红绿色盲友好）
- **i18n v1 范围**：UI 仅 zh-CN；README 双语（zh-CN 主、en 摘要）
- **i18n v1.1 计划**：zh-TW、zh-HK 繁体；en UI；UI 字符串走 `i18next` 或自写 lookup（看 bundle size 取舍）
- **屏幕阅读器**：`aria-label` 标注图表数据点；表单走 `<label for>` 标准；动态内容用 `aria-live="polite"`

## Data Integrity & Validation

- **输入边界**（zod schema 强制于 core 和 mcp 入口）：
  - `ageMonths`: 0–228（>228 抛 `OutOfRangeError`；0 和负数抛错）
  - `heightCm`: 30–250（异常抛 `OutOfPlausibleRangeError`，不静默继续）
  - `weightKg`: 0.5–200
  - `headCircumferenceCm`: 25–70
  - 不合理组合（如 6 个月 80kg）：core 不做相互校验（避免误杀有效数据），但 web 端弹"测量值看起来异常，确认输入？"软提示
- **算法精度**：core 输出的 z-score 必须 |delta| ≤ 0.01 vs R `WorldHealthOrganization/anthro`；CI 跑 golden 对照测试，超出容差则 fail
- **临床免责**：所有 `assess` / `interpret` 返回都强制带 `disclaimer` 字段；MCP 工具响应同步包含

## Open Questions

1. **卫健委 2023 LMS 数据怎么拿？**——官方只发 PDF。两条路并行尝试：
   (a) 检查 R 包 `childsds` 和 `WHO/anthro` 仓库的 `data-raw/` 看 NHC 数据是否被收录
   (b) `pdfplumber`（Python）或 `tabula-java` 提取 PDF 表 → CSV → 人工核校
   **这是 v1 的最大单点风险**；Week 1 第一项工作就是攻克这个。
2. **图表库：手写 SVG vs uPlot？**——倾向手写（5 条线 + 数据点，复杂度低；零依赖；core 中跑得动）。Week 3 第一天验证。
3. **PWA 化是否 v1 做？**——v1 想做但有时间风险；保底 v1.1 补。
4. **MCP transport：stdio 是否够用？**——v1 只 stdio。HTTP 留 v1.1。
5. **多孩子 v1 是否支持？**——是。schema 已设计成 multi-child（见上）。
6. **Web 端是否嵌入 MCP playground？**——v2 考虑。
